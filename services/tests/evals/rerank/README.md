# Reranker model-selection study

Finds a small, on-device model to replace the 7-10 GB MLX classifier for matching a
coding-agent **session summary** to a PM ticket — deciding both *which* ticket the work
advances and *whether it matches none* (→ create-new). Scenario mirrors KAN-239: the
developer confirms a 2-4 task daily plan, so each session is scored against only those
few candidates.

## Result

| Model | Accuracy | False-binds | RAM | Notes |
|---|---|---|---|---|
| **Qwen3-Reranker-0.6B-MLX-4bit** | **94%** | 0 | 1.5 GB | lean choice, instruction-tuned yes/no gate |
| 0.6B + 4B **MAX** ensemble | **96%** | 0 | 3.3 GB | max-accuracy; 4B vetoes 0.6B recall misses |
| Qwen3-Reranker-4B-4bit | 91% | 0 | 3.3 GB | bigger ≠ better here |
| bge-reranker-v2-m3 | 75% | 11 | 3.5 GB | RAG ranker, no clean abstain gate |
| gte-reranker-modernbert-base | 53% | 38 | 1.4 GB | saturated, binds everything |

Operating point: bind to argmax candidate if its score ≥ ~0.08-0.10, else create-new.
Residual misses are recall on the genuinely-overlapping KAN-231/240/241 ticket cluster
(several labels contestable); on unambiguous sessions the ensemble is ≈99%.

## Dataset (`data/`)

- `sessions_all.json` — 85 real Claude Code coding-agent session summaries (06-16..06-19).
- `tasks.json` — 25-ticket pool: 22 open `pm_tasks` + KAN-231/239/240 (now Done) with Jira descriptions.
- `labels_all.py` — hand-labeled ground truth: `{session_id: (primary, acceptable_set, uncertain_flag, note)}`.

Rebuild from `meridian.db` (sessions + open tasks) with `python build_data.py`.

## Harness

- `plans.py` — shared daily-plan builder (seeded): per session, a 2-3 candidate plan = true ticket + diverse distractors (or all-distractors for create-new cases). Every model sees identical inputs.
- `eval_mlx.py <repo>` — MLX Qwen3-Reranker (yes/no logit → calibrated score). Default = 0.6B.
- `eval_ce.py <repo>` — BERT-style cross-encoders (bge / gte) via `sentence-transformers`.
- `techniques.py` — accuracy-lift sweep on the 0.6B: doc representation, instruction variants, instruction ensemble.
- `combine_final.py` — offline 0.6B+4B ensemble (mean/max) + residual-miss characterization.
- `results/` — cached per-model per-candidate scores (regenerated by the eval scripts).

## Running

```bash
# MLX rerankers (use the services venv — has mlx_lm):
source ../../../.venv/bin/activate
python eval_mlx.py                                   # 0.6B, 94%
python eval_mlx.py vserifsaglam/Qwen3-Reranker-4B-4bit-MLX
python techniques.py
python combine_final.py                              # ensemble (reads results/)

# BERT-style cross-encoders need torch (separate venv, not the services venv):
python -m venv .venv && .venv/bin/pip install torch sentence-transformers
.venv/bin/python eval_ce.py BAAI/bge-reranker-v2-m3
```

Each script loads exactly one model and exits, so memory frees between runs (no two
models resident at once). `.venv/` here is the throwaway torch env — gitignore it.
